Section 01
EC2 구매 옵션 (Purchasing Options)
리조트 호텔에 비유하면 각 옵션의 차이가 한눈에 들어옵니다.
🏨 On-Demand 기준 요금

온디맨드 인스턴스

"원할 때 리조트에 와서, 정가 그대로 내고 머무는 것 — 예약 없이 체크인, 언제든 체크아웃."

  • ›약정 없음, 초 단위 or 시간 단위 과금
  • ›단기·불규칙 워크로드에 적합
  • ›처음 AWS를 써보거나 테스트할 때 추천
  • ›Linux: 초 단위 / Windows: 시간 단위
📅 Reserved 최대 72% 할인

예약 인스턴스

"미리 1년~3년 예약하면 큰 할인 — 체크인 날짜가 정해졌다면 예약이 유리."

  • ›약정기간: 1년 또는 3년 (3년이 더 큰 할인)
  • ›Standard RI: 최대 72%, 변경 불가, Marketplace 판매 가능
  • ›Convertible RI: 최대 66%, 인스턴스 패밀리/OS 변경 가능
  • ›DB, 안정적 장기 워크로드에 최적
💳 Savings Plans 최대 66% 할인

세이빙스 플랜

"시간당 일정 금액 약정, 어떤 방 종류든 OK (킹룸, 스위트, 오션뷰) — 방 타입을 자유롭게 바꿀 수 있음."

  • ›약정기간: 1년 또는 3년
  • ›Compute Savings: 모든 인스턴스 패밀리·리전 유연 사용
  • ›EC2 Instance Savings: 특정 패밀리 내 유연 사용 (최대 72%)
  • ›Reserved보다 유연하지만 할인율은 동등
⚡ Spot 최대 90% 할인

스팟 인스턴스

"빈 방에 입찰 — 최고가 입찰자가 방을 차지하지만, 호텔이 더 높은 가격 손님이 오면 당신을 언제든 내보낼 수 있음."

  • ›가장 큰 할인이지만 중단 위험 존재
  • ›AWS가 필요 시 2분 경고 후 강제 종료
  • ›중단 가능한 배치 작업, ML 학습, 데이터 분석에 적합
  • ›중요 DB, 프로덕션 서버에는 절대 사용 금지
🏢 Dedicated Hosts RI 시 최대 70%

전용 호스트

"리조트의 건물 전체를 통째로 빌리는 것 — 누가 몇 층 몇 호에 있는지 전부 내가 관리."

  • ›물리적 서버 전체를 단독 사용
  • ›소켓·코어·VM 가시성 확보 → BYOL 라이선스에 필수
  • ›엄격한 규정 준수(Compliance) 요구 사항 충족
  • ›On-Demand 또는 Reserved(1·3년) 방식으로 구매
📦 Capacity Reservations 할인 없음

온디맨드 용량 예약

"방을 예약해두되, 실제로 안 자더라도 방값은 내야 함 — 언제든 쓸 수 있게 방을 붙잡아두는 것."

  • ›특정 AZ의 On-Demand 용량 보장
  • ›약정 없음, 언제든 생성·취소 가능
  • ›사용 여부와 무관하게 On-Demand 요금 발생
  • ›Savings Plans / RI와 결합해 비용 절감 가능
구매 옵션 비교표
옵션 할인율 약정기간 인스턴스 중단 위험 결제 방식 주요 사용 사례
On-Demand
온디맨드 		기준 요금 없음 없음 사용한 만큼 단기, 불규칙, 테스트
Standard RI
표준 예약 		최대 72% 1년 or 3년 없음 No / Partial / All Upfront DB, 안정적 장기 서버
Convertible RI
전환형 예약 		최대 66% 1년 or 3년 없음 No / Partial / All Upfront 인스턴스 변경이 필요한 장기 워크로드
Savings Plans
세이빙스 플랜 		최대 66~72% 1년 or 3년 없음 시간당 금액 약정 다양한 인스턴스 유연 사용
Spot
스팟 		최대 90% 🔥 없음 있음 ⚠️ 사용한 만큼 (경매가) 배치, ML 학습, 비중요 작업
Dedicated Hosts
전용 호스트 		RI 적용 시 최대 70% 없음 / 1·3년 선택 없음 On-Demand or RI BYOL, 규정 준수, 소켓/코어 가시성
Dedicated Instance
전용 인스턴스 		약간 높은 요금 없음 없음 사용한 만큼 하드웨어 물리적 격리 (소켓 가시성 없음)
Capacity Reservations
용량 예약 		없음 없음 (언제든 취소) 없음 On-Demand 요금 (미사용도 과금) 특정 AZ 용량 확보 필요 시
💡
Standard RI vs Convertible RI 핵심 차이: Standard는 할인율이 더 크지만(72%) 인스턴스 패밀리를 바꿀 수 없고, Marketplace에서 타인에게 판매할 수 있습니다. Convertible는 인스턴스 패밀리·OS·테넌시를 변경할 수 있지만 Marketplace 판매가 불가합니다. 시험에서 "가장 큰 할인 + 변경 불가"가 나오면 Standard RI, "변경 유연성"이 나오면 Convertible RI를 선택하세요.
⚠️
Spot 인스턴스 중단 시나리오: AWS가 스팟 인스턴스를 회수할 경우 2분 전 경고를 줍니다. 이 2분 안에 체크포인트 저장 등의 종료 처리를 해야 합니다. DB, 결제 시스템처럼 중단되면 안 되는 서비스에는 절대 사용하지 마세요.
Section 02
보안 그룹 (Security Groups)
EC2 인스턴스의 가상 방화벽 — 인바운드/아웃바운드 트래픽을 제어합니다.
🌐 인터넷 (Internet)
↓ Inbound
↑ Outbound
🛡️ Security Group (가상 방화벽)
EC2
Instance A
EC2
Instance B
하나의 보안 그룹 → 여러 인스턴스에 적용 가능

🔵 Inbound (인바운드) 규칙

기본값: 모든 인바운드 차단
명시적으로 허용 규칙을 추가해야 접근 가능

허용 규칙만 설정 가능 (Deny 규칙 없음)

예: SSH 22번 포트를 내 IP에서만 허용

🟢 Outbound (아웃바운드) 규칙

기본값: 모든 아웃바운드 허용
인터넷으로 나가는 트래픽은 기본 열림

Stateful: 인바운드를 허용하면 응답 트래픽은 자동 허용

NACL(Network ACL)과 달리 응답 규칙을 별도로 안 써도 됨

📌 보안 그룹 핵심 특성

하나의 SG → 여러 EC2에 적용 가능
반대로, 하나의 EC2에 여러 SG를 붙일 수 있음

특정 리전 / VPC에 종속 (다른 리전에서 재사용 불가)

다른 보안 그룹을 Source로 참조 가능 (IP 대신 SG 지정)

⚡ 타임아웃 vs 연결 거부

타임아웃 발생 시 → 보안 그룹 문제
포트가 막혀 있거나 허용 규칙이 없는 것

Connection Refused → 앱 문제
보안 그룹은 통과했지만 앱이 응답 안 함

EC2와는 달리 보안 그룹은 EC2 외부에서 동작 (EC2가 뭘 받는지 모름)

🧠
Stateful 기억법: 보안 그룹은 "대화를 기억"합니다. 내가 먼저 HTTP 요청을 허용했다면, 그 응답은 자동으로 통과시킵니다. 반면 NACL은 Stateless — 요청과 응답을 각각 별도로 허용해야 합니다. (DVA-C02에서 NACL vs SG 비교 문제 자주 출제!)
Section 03
EC2 인스턴스 유형 (Instance Types)
시험에서 "어떤 인스턴스 유형?" 문제는 키워드로 판단합니다 — 각 유형의 핵심 키워드를 외워두세요.
General Purpose

범용 인스턴스

M5, M6g, T2, T3, T4g

CPU, 메모리, 네트워크 균형 잡힌 리소스. 웹 서버, 코드 리포지토리, 소규모 DB 등 다양한 워크로드에 적합.

⚡ 시험 키워드: "균형 / 다목적 / 웹 서버"
Compute Optimized

컴퓨팅 최적화

C5, C5n, C6g, C6gn

고성능 프로세서를 사용하는 컴퓨팅 집약적 워크로드에 적합. 고성능 컴퓨팅(HPC), 과학 시뮬레이션, ML 추론, 게임 서버, 미디어 트랜스코딩.

⚡ 시험 키워드: "HPC / 고성능 컴퓨팅 / C로 시작"
Memory Optimized

메모리 최적화

R6g, R5, X1, X2, z1d

메모리에서 대용량 데이터를 처리하는 워크로드. 인메모리 DB(Redis, Memcached), 실시간 빅데이터, SAP HANA, 고성능 데이터 분석.

⚡ 시험 키워드: "인메모리 / RAM 집약적 / 실시간 처리"
Storage Optimized

스토리지 최적화

I3, I4i, D2, D3, H1

로컬 스토리지에 매우 빠르고 빈번하게 읽기/쓰기하는 워크로드. 고빈도 OLTP 데이터베이스, 분산 파일시스템, 데이터 웨어하우징, NoSQL DB.

⚡ 시험 키워드: "OLTP / 고빈도 I/O / 초당 수천 건"
Accelerated Computing

가속 컴퓨팅

P3, P4, G4, G5, F1

하드웨어 가속기(GPU, FPGA)를 사용하는 워크로드. ML 학습, 그래픽 처리, 게임 스트리밍, 비디오 렌더링, FPGA 기반 연산.

⚡ 시험 키워드: "GPU / ML 학습 / 그래픽 / FPGA"
유형 패밀리 핵심 리소스 시험 키워드
General Purpose M, T 균형형 (CPU+RAM+Network) 웹 서버, 다목적, 균형
Compute Optimized C 고성능 CPU HPC, 고성능 컴퓨팅, 게임 서버
Memory Optimized R, X, z 고용량 RAM 인메모리 DB, 실시간 빅데이터
Storage Optimized I, D, H 고성능 NVMe/SSD OLTP, 고빈도 I/O, NoSQL
Accelerated Computing P, G, F GPU / FPGA ML 학습, 그래픽, FPGA
Section 04
시험 예상 문제 Q&A
클릭해서 정답과 해설을 확인하세요. 실제 시험 문제 패턴과 동일합니다.
1
가장 큰 할인을 제공하지만 중요한 작업이나 데이터베이스에는 적합하지 않은 EC2 구매 옵션은?
▶
정답
⚡ Spot Instance (스팟 인스턴스)
스팟 인스턴스는 최대 90% 할인으로 EC2 구매 옵션 중 가장 저렴합니다. 그러나 AWS가 해당 용량을 필요로 하면 2분 경고 후 인스턴스를 강제 종료시킵니다. 따라서 중단되어도 괜찮은 배치 작업, ML 학습, 데이터 분석에는 적합하지만 DB, 결제 시스템, 프로덕션 서버에는 절대 사용하면 안 됩니다.
🎯 시험 팁 "가장 큰 할인" + "중단 가능성" = Spot Instance. 이 두 키워드가 함께 나오면 즉시 Spot을 선택하세요.
2
EC2 인스턴스 안팎의 트래픽을 제어하려면 무엇을 사용해야 할까요?
▶
정답
🛡️ Security Group (보안 그룹)
보안 그룹은 EC2 인스턴스에 대한 가상 방화벽입니다. 인바운드 규칙(외부 → EC2)과 아웃바운드 규칙(EC2 → 외부)을 설정해 허용할 트래픽을 세밀하게 제어합니다. 기본적으로 모든 인바운드는 차단, 모든 아웃바운드는 허용됩니다.
💡 참고 VPC 레벨의 트래픽 제어는 NACL(Network ACL)이 담당합니다. 보안 그룹은 Stateful, NACL은 Stateless입니다.
3
EC2 예약 인스턴스(Reserved Instance)를 얼마나 길게 예약할 수 있습니까?
▶
정답
📅 최대 3년 (1년 또는 3년 선택)
EC2 예약 인스턴스는 1년 또는 3년 중에서 선택합니다. 약정 기간이 길수록 할인율이 높아집니다. 3년 약정 + All Upfront 결제 시 최대 72% 할인을 받을 수 있습니다.
🎯 시험 팁 선택지에 "6개월"이 보이면 함정입니다. RI는 1년 또는 3년만 존재합니다.
4
EC2에 고성능 컴퓨팅(HPC) 애플리케이션을 배포하려고 합니다. 어떤 인스턴스 유형을 선택해야 할까요?
▶
정답
🖥️ Compute Optimized (컴퓨팅 최적화 — C 패밀리)
컴퓨팅 최적화 인스턴스(C5, C6g 등)는 고성능 프로세서를 사용해 CPU 집약적인 작업에 최적화되어 있습니다. HPC, 과학 시뮬레이션, 게임 서버, ML 추론, 미디어 트랜스코딩이 대표적인 사용 사례입니다.
🎯 시험 팁 "HPC" = Compute Optimized (C 패밀리). 절대 General Purpose와 헷갈리지 마세요.
5
1년 동안 서버에서 지속해서 실행할 계획인 애플리케이션에 적합한 EC2 구매 옵션은 무엇일까요?
▶
정답
📅 Reserved Instance (예약 인스턴스)
1년 이상 지속적으로 실행되는 예측 가능한 워크로드라면 예약 인스턴스(RI)가 가장 비용 효율적입니다. On-Demand 대비 최대 72% 할인을 받을 수 있으며, 안정적인 DB 서버나 상시 운영 서버에 이상적입니다.
🎯 시험 팁 "1년 이상 지속" + "예측 가능한 워크로드" = Reserved Instance. Spot은 중단 위험 때문에 오답입니다.
6
EC2 인스턴스를 시작할 때 소프트웨어 설치와 OS 패키지 업데이트를 자동으로 수행하는 가장 좋은 방법은?
▶
정답
⚙️ EC2 User Data (유저 데이터 스크립트)
User Data는 EC2 인스턴스가 최초 시작될 때 단 한 번 자동 실행되는 부트스트랩 스크립트입니다. Root 권한으로 실행되며, 패키지 설치·서비스 시작·파일 다운로드 등 모든 초기화 작업을 자동화할 수 있습니다.
#!/bin/bash # 인스턴스 최초 시작 시 자동 실행 (Bootstrap Script) yum update -y yum install -y httpd systemctl start httpd systemctl enable httpd echo "Hello from EC2" > /var/www/html/index.html
🎯 시험 팁 "인스턴스 시작 시 자동 설치/설정" 키워드가 나오면 무조건 User Data. AMI와 헷갈리지 마세요 — AMI는 이미지 자체를 커스텀하는 것이고, User Data는 런타임 초기화입니다.
7
인메모리 데이터베이스(In-Memory DB)를 사용하는 중요 애플리케이션을 위해서는 어떤 EC2 인스턴스 유형을 선택해야 할까요?
▶
정답
🧠 Memory Optimized (메모리 최적화 — R/X 패밀리)
인메모리 DB(Redis, Memcached, SAP HANA)는 데이터를 RAM에 올려두고 처리하기 때문에 대용량 고속 메모리가 필수입니다. R 패밀리(R5, R6g)나 X 패밀리(X1, X2)는 다른 인스턴스 유형 대비 훨씬 많은 메모리를 제공합니다.
🎯 시험 팁 "인메모리" = Memory Optimized (R 패밀리). R은 RAM의 R이라고 기억하세요!
8
데이터베이스에 초당 수천 개의 요청을 처리하는 고빈도 OLTP 데이터베이스를 EC2로 마이그레이션하려고 합니다. 어떤 인스턴스 유형을 선택해야 할까요?
▶
정답
💾 Storage Optimized (스토리지 최적화 — I/D 패밀리)
OLTP(Online Transaction Processing)는 매 초 수천~수만 건의 읽기/쓰기가 발생하는 워크로드입니다. Storage Optimized 인스턴스(I3, I4i)는 고성능 NVMe SSD를 탑재해 낮은 지연시간과 높은 IOPS(초당 I/O 작업 수)를 제공합니다.
🎯 시험 팁 "OLTP" + "고빈도 I/O" + "초당 수천 건" = Storage Optimized (I 패밀리). Memory와 헷갈리지 마세요 — 메모리가 아닌 디스크 속도가 핵심입니다.
9
"보안 그룹은 하나의 EC2 인스턴스에만 연결할 수 있다." — 이 문장은 참인가요, 거짓인가요?
▶
정답
❌ 거짓 (False)
보안 그룹은 여러 EC2 인스턴스에 동시에 연결할 수 있습니다. 반대로, 하나의 EC2 인스턴스에 여러 보안 그룹을 부착하는 것도 가능합니다. 예를 들어 "Web-SG"를 웹 서버 3대 모두에 적용하고, 특정 서버에만 "Admin-SG"를 추가로 붙일 수 있습니다.
💡 정리 보안 그룹 ↔ EC2 관계는 다대다(N:M)입니다. 1개의 SG → 여러 EC2 가능 / 1개의 EC2 → 여러 SG 가능.
10
온프레미스 앱을 AWS로 마이그레이션할 때 애플리케이션이 전용 서버에서 실행되어야 하는 규정 준수 요구 사항이 있고, 자체 소프트웨어 라이선스(BYOL)를 사용해야 합니다. 어떤 EC2 구매 옵션이 적합할까요?
▶
정답
🏢 Dedicated Hosts (전용 호스트)
Dedicated Hosts는 물리적 서버 전체를 단독으로 할당받습니다. 이를 통해 규정 준수(Compliance) 요구사항을 충족하고, 소켓·코어 단위로 라이선스가 책정되는 Oracle, SQL Server, Windows Server 등의 BYOL(Bring Your Own License) 소프트웨어를 사용할 수 있습니다.
🎯 시험 팁 "전용 서버 규정 준수" + "BYOL" = Dedicated Hosts. Dedicated Instance와 혼동 금지 — Instance는 소켓/코어 가시성이 없어 BYOL에 부적합합니다.
11
공급업체 라이선스가 물리적 코어 및 기본 네트워크 소켓 가시성을 기반으로 비용을 청구합니다. 어떤 EC2 구매 옵션으로 이 가시성을 확보할 수 있을까요?
▶
정답
🔍 Dedicated Hosts (전용 호스트)
Dedicated Hosts는 물리적 서버의 소켓 수, 코어 수, 하드웨어 ID를 직접 확인할 수 있습니다. 일부 소프트웨어 라이선스(Oracle Database, Microsoft 등)는 물리적 코어나 소켓 수 기반으로 비용을 책정하기 때문에 이 정보가 반드시 필요합니다.

반면 Dedicated Instance는 물리적 하드웨어를 단독 사용하지만 소켓·코어 등의 하드웨어 세부 정보를 볼 수 없습니다.
💡 Dedicated Hosts vs Dedicated Instance 핵심 차이 Dedicated Hosts: 물리적 서버 단독 사용 + 소켓/코어 가시성 O + BYOL 가능 Dedicated Instance: 하드웨어 격리 O + 소켓/코어 가시성 X + BYOL 불가
🎯 시험 팁 "물리적 코어/소켓 가시성" 키워드가 보이면 → 무조건 Dedicated Hosts!
Section 05
빠른 요약 & 시험 TIP
시험 직전에 훑어볼 핵심 요약과, 헷갈리는 키워드-정답 연결고리 모음입니다.
⚡ 빠른 요약
💰 가장 큰 할인 = Spot
최대 90% 할인이지만 AWS가 언제든 2분 경고 후 강제 종료 — DB/중요 서버 절대 금지
📅 RI 최대 약정기간 = 3년
1년 또는 3년만 선택 가능. 6개월은 존재하지 않음. 3년 All Upfront 시 최대 72% 할인
🏢 BYOL + 물리 코어 = Dedicated Hosts
소켓·코어 가시성이 있어 라이선스 비용 계산 가능. Dedicated Instance는 가시성 없음
🛡️ 보안 그룹 = N:M 관계
1개 SG → 여러 EC2 / 1개 EC2 → 여러 SG. "하나에만 연결" 선택지는 항상 오답
⚙️ 시작 시 자동 설치 = User Data
인스턴스 최초 부팅 시 1회 실행되는 Bootstrap 스크립트. Root 권한으로 동작
🔌 타임아웃 vs 연결 거부
타임아웃 → 보안 그룹 문제 / Connection Refused → 앱 문제 (SG는 통과된 것)
🎯 시험 TIP — 키워드 → 정답 연결고리
  • 가장 큰 할인 + 중단 가능성 → Spot Instance
  • 1년 이상 지속 실행 + 예측 가능한 워크로드 → Reserved Instance
  • 다양한 인스턴스 패밀리를 유연하게 전환 → Savings Plans (Compute)
  • BYOL + 전용 서버 규정 준수 + 소켓/코어 가시성 → Dedicated Hosts
  • HPC / 고성능 컴퓨팅 / 게임 서버 → Compute Optimized (C 패밀리)
  • 인메모리 DB / RAM 집약적 → Memory Optimized (R 패밀리)
  • OLTP / 고빈도 I/O / 초당 수천 건 DB 요청 → Storage Optimized (I 패밀리)
  • ML 학습 / GPU / FPGA → Accelerated Computing (P·G·F 패밀리)
  • 인스턴스 시작 시 소프트웨어 자동 설치 → EC2 User Data (Bootstrap Script)
  • EC2 트래픽 제어 (인바운드/아웃바운드) → Security Group (보안 그룹)
  • 보안 그룹 Stateful vs NACL Stateless → SG는 응답 자동 허용 / NACL은 요청·응답 각각 허용 필요
