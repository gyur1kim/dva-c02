Storage 101

그래서 "스토리지"가 뭔데?

스토리지(storage)는 한마디로 데이터를 저장해두는 공간이야. EC2가 "일하는 컴퓨터(CPU + 메모리)"라면, 스토리지는 그 컴퓨터가 파일·데이터를 올려두고 읽고 쓰는 디스크에 해당해.

🧠
프론트 개발자 비유: 브라우저에도 데이터를 저장하는 곳이 계층별로 있잖아 — 메모리 변수(껐다 켜면 사라짐) → sessionStorage(탭 닫으면 사라짐) → localStorage / IndexedDB(영구). AWS 스토리지도 정확히 이 "얼마나 오래 남는가 + 누가 접근 가능한가" 축으로 나뉜다고 보면 이해가 빨라.

스토리지를 나누는 3가지 기준

💾 저장 방식

블록 vs 파일 vs 객체

  • ›블록: OS가 "디스크"로 인식 (EBS, Instance Store) — DB·부팅용
  • ›파일: 폴더·파일 구조로 공유 (EFS) — NFS 마운트
  • ›객체: 통째로 넣고 URL로 꺼냄 (S3) — 이번 범위 밖
⏳ 지속성

영구 vs 임시

"localStorage냐 메모리 변수냐"

  • ›영구(persistent): 인스턴스 꺼져도 데이터 유지 (EBS, EFS)
  • ›임시(ephemeral): 인스턴스 종료 시 소실 (Instance Store)
🌐 접근 범위

1대 vs 여러 대

  • ›단일 인스턴스 전용 (EBS 기본, Instance Store)
  • ›여러 인스턴스·여러 AZ 동시 공유 (EFS)
  • ›이 축이 시험 정답을 가장 많이 가른다
💡
이번 가이드의 핵심: EBS · Instance Store · EFS는 결국 위 3가지 기준에서 어디에 위치하느냐만 다른 거야. 아래에서 하나씩 살펴보고, 마지막에 한 표로 비교할게.
Elastic Block Store

EBS 핵심 개념

EBS는 EC2에 붙이는 네트워크 드라이브야. 물리적으로 붙은 게 아니라 네트워크로 연결되기 때문에 인스턴스와 분리·재연결이 가능해. 대신 이 특성 때문에 AZ에 묶인다는 제약이 생겨.

📍 AZ 고정

단일 AZ에 종속

"us-east-1a의 볼륨은 us-east-1a의 인스턴스에만"

  • ›다른 AZ 인스턴스에 직접 연결 불가
  • ›AZ 이동하려면 스냅샷 → 다른 AZ에서 복원
  • ›스냅샷은 리전 간 복사도 가능
🗑 종료 동작

DeleteOnTermination

"루트는 지워지고, 추가 볼륨은 남는다"

  • ›루트 볼륨: 기본값 true (삭제)
  • ›추가 EBS: 기본값 false (보존)
  • ›콘솔에서 이 플래그 변경 가능
🔗 Multi-Attach

다중 연결

"io1/io2만, 같은 AZ, 최대 16대"

  • ›하나의 볼륨을 여러 EC2에 동시 연결
  • ›io1 / io2 유형에서만 지원
  • ›같은 AZ 내 최대 16 인스턴스
  • ›클러스터 인식 파일시스템 필요
💡
가장 헷갈리는 함정: "다른 AZ의 인스턴스에 연결이 안 된다" → 문제 상황이 아니라 정상 동작이야. EBS는 태생적으로 AZ에 묶여 있어. 이동하려면 스냅샷을 떠서 다른 AZ에서 복원해야 해.
EBS Volume Types

EBS 볼륨 유형 비교

시험 단골. 부팅 볼륨 가능 여부 = SSD냐 HDD냐로 갈려. SSD 계열(gp·io)만 부팅 볼륨이 될 수 있고, HDD 계열(st1·sc1)은 안 돼.

유형분류최대 IOPS부팅 볼륨주 용도
gp3SSD (범용)16,000 ✔ 가능기본 추천 · IOPS/처리량 독립 설정
gp2SSD (범용)16,000 ✔ 가능구세대 범용 · 크기 = IOPS (3/GB)
io1SSD (고성능)64,000 ✔ 가능고성능 DB · Multi-Attach 지원
io2 Block ExpressSSD (최고성능)256,000 ✔ 가능초고성능 DB · Multi-Attach 지원
st1HDD (처리량)500 ✘ 불가빅데이터 · 로그 · 데이터 웨어하우스
sc1HDD (콜드)250 ✘ 불가접근 빈도 낮은 데이터 · 최저 비용
⚠️
gp2 IOPS 캡 함정: gp2는 3 IOPS/GB지만 최대 16,000 IOPS에서 잘려. 약 5,334GB(≈5.3TB)면 이미 최대치야. 그래서 8TB gp2는 이미 16,000 캡 상태라 크기를 더 늘려도 IOPS가 안 올라가. IOPS를 더 원하면 gp3나 io1/io2로 마이그레이션해야 해.
Amazon Machine Image

AMI

AMI는 EC2를 띄우는 템플릿(OS + 설정 + 앱 사전 패키징). 프론트로 치면 프로젝트를 부트스트랩하는 보일러플레이트 템플릿이랑 비슷해. 중요한 건 AMI는 리전 단위라는 점이야.

🌎 리전 종속

AMI는 리전에 묶인다

  • ›만든 리전에서만 인스턴스 시작 가능
  • ›다른 리전에서 쓰려면 AMI 복사(Copy)
  • ›복사하면 새 리전에 새 AMI ID 생성
⚙️ 구성

AMI에 담기는 것

  • ›루트 볼륨의 EBS 스냅샷
  • ›실행 권한(누가 launch 가능한지)
  • ›블록 디바이스 매핑
🧠
기억법: "us-east-1의 AMI로 모든 리전에서 시작 가능?" → 거짓. AMI = 리전 스코프. 다른 리전은 복사 먼저. (참고: EBS 스냅샷도 리전 단위, EBS 볼륨은 AZ 단위로 스코프가 계단식으로 커진다고 외우면 편해.)
EC2 Instance Store

EC2 Instance Store

호스트 서버에 물리적으로 붙은 로컬 디스크. 네트워크를 안 타서 초고성능(수백만 IOPS)이지만, 인스턴스가 종료/중지되면 데이터가 사라지는 임시(ephemeral) 저장소야.

⚡ 초고성능

물리적 로컬 디스크

"네트워크를 안 거치는 직결 SSD"

  • ›EBS보다 훨씬 높은 IOPS/처리량
  • ›수십만~수백만 IOPS 가능
  • ›버퍼·캐시·스크래치 데이터에 최적
💨 임시

종료 시 데이터 소실

"인스턴스와 운명 공동체"

  • ›인스턴스 종료·중지 시 데이터 손실
  • ›영구 저장에는 부적합
  • ›백업/복제는 직접 책임져야 함
Elastic File System

EFS

관리형 NFS 파일 시스템. EBS와 달리 여러 AZ에 걸친 수백~수천 개 EC2가 동시에 같은 데이터를 마운트할 수 있어. "공유 파일 스토리지"가 필요하면 정답은 거의 EFS야.

🔀 다중 AZ 공유

NFS로 동시 마운트

"여러 AZ · 여러 EC2 → 하나의 파일시스템"

  • ›NFS 프로토콜로 마운트
  • ›수백~수천 인스턴스 동시 접근
  • ›여러 AZ 걸쳐 공유 가능
📈 자동 확장

용량 자동 스케일

  • ›용량 프로비저닝 불필요 (사용한 만큼)
  • ›Linux 전용 (Windows 미지원)
  • ›gp2 대비 약 3배 비용 (스토리지 클래스로 절감)
🧠
키워드 매칭: "여러 AZ" + "동일 데이터 공유" + "NFS" → EFS. 반대로 "하나의 인스턴스" + "부팅/DB" → EBS. "임시 고성능 캐시" → Instance Store.
Head to Head

EBS vs Instance Store vs EFS

이 표 하나만 확실히 외우면 이번 주 문제의 절반은 바로 풀려. 특히 지속성 / AZ 범위 / 동시 연결 세 열이 정답을 가른다.

💡
한 줄 구분: EBS=한 인스턴스 전용 영구 드라이브(AZ 고정) · Instance Store=빠르지만 사라지는 임시 로컬 디스크 · EFS=여러 AZ가 나눠 쓰는 공유 파일함.
Practice Questions

시험 예상 문제 Q&A

이번에 어렵다고 한 9문제. 클릭해서 정답·해설·시험 팁 확인해봐.

1
us-east-1a에서 EC2를 종료했고 연결됐던 EBS 볼륨을 사용할 수 있게 됨. 팀원이 us-east-1b의 EC2에 연결하려는데 안 됨. 원인은?
▶
정답
📍 EBS 볼륨은 단일 AZ에 묶여 있기 때문
EBS 볼륨은 생성된 AZ 안에서만 인스턴스에 붙일 수 있어. us-east-1a의 볼륨은 us-east-1b의 인스턴스에 직접 연결 불가. 옮기려면 스냅샷을 만들고 → us-east-1b에서 그 스냅샷으로 볼륨을 복원해서 붙여야 해.
🎯 시험 팁 "다른 AZ에 연결이 안 된다"는 버그가 아니라 정상 동작. 이동 방법을 묻는 문제엔 항상 스냅샷이 정답 키워드.
2
루트 볼륨 + 추가 EBS 볼륨에 데이터 저장. 한 달 뒤 인스턴스를 종료할 때, 각 EBS 볼륨의 기본 동작은?
▶
정답
🗑 루트=삭제, 추가 볼륨=보존
루트 볼륨은 DeleteOnTermination = true가 기본이라 종료 시 같이 삭제돼. 반면 나중에 붙인 추가 EBS 볼륨은 DeleteOnTermination = false가 기본이라 인스턴스가 사라져도 볼륨은 남아. (두 플래그 모두 설정으로 변경 가능)
🎯 시험 팁 "루트는 지워지고 데이터 볼륨은 살아남는다"로 외우기. 반대로 "종료해도 루트 데이터 유지하려면?" → DeleteOnTermination을 false로.
3
us-east-1(N.Virginia)의 AMI를 사용해 모든 AWS 리전에서 EC2를 시작할 수 있다. (True / False)
▶
정답
❌ 거짓AMI는 리전 단위
AMI는 만든 리전에서만 인스턴스를 시작할 수 있어. 다른 리전에서 쓰려면 먼저 AMI를 대상 리전으로 복사(Copy AMI)해야 하고, 복사하면 새 리전에 새 AMI ID가 생성돼.
🎯 시험 팁 스코프 계단: EBS 볼륨=AZ · 스냅샷/AMI=리전 · IAM=글로벌. "cross-region"이 나오면 AMI는 Copy 필요.
4
EC2 인스턴스를 생성할 때 부팅(루트) 볼륨으로 쓸 수 있는 EBS 볼륨 유형은?
▶
정답
💽 SSD 계열 — gp2 / gp3 / io1 / io2
부팅 볼륨이 되려면 SSD 계열이어야 해. gp2, gp3, io1, io2는 가능하고, HDD 계열인 st1·sc1은 부팅 볼륨 불가. HDD는 순차 처리량용(빅데이터·로그)이라 OS 부팅에는 못 써.
🎯 시험 팁 "boot volume 가능한 유형?" → SSD만(gp·io). st1/sc1이 보기에 있으면 그게 함정 오답.
5
EBS 다중 연결(Multi-Attach)이란 무엇인가?
▶
정답
🔗 하나의 볼륨을 같은 AZ 내 여러 EC2에 동시 연결
단일 io1/io2 볼륨을 같은 AZ 안에서 최대 16개 EC2 인스턴스에 동시에 붙이는 기능. 클러스터형 Linux 애플리케이션(고가용성 처리)에서 써. 여러 인스턴스가 동시에 쓰기 때문에 클러스터 인식 파일시스템이 필요하고, 일반 XFS/EXT4는 부적합.
🎯 시험 팁 Multi-Attach = io1/io2 전용 · 같은 AZ · 최대 16대. "gp2로 Multi-Attach" 같은 보기는 무조건 오답.
6
8TB gp2 볼륨을 프로비전했는데 IOPS가 부족하다. 성능을 높이는 방법이 아닌 것은?
▶
정답 (방법이 아닌 것)
🚫 gp2 볼륨 크기를 더 늘리기
gp2는 3 IOPS/GB지만 최대 16,000 IOPS에서 캡돼. 16,000 ÷ 3 ≈ 5,334GB(약 5.3TB)에서 이미 최대치에 도달하거든. 8TB(8,000GB)는 이미 16,000 IOPS 캡 상태라 크기를 더 키워도 IOPS가 안 올라가. 그래서 "크기 늘리기"는 성능 향상 방법이 아니야.

대신 gp3로 마이그레이션(크기와 무관하게 IOPS 별도 설정) 하거나 io1/io2로 마이그레이션(더 높은 IOPS)하는 게 올바른 방법이야.
🤔
gp2가 아니라 gp3였어도 정답이 같을까? — 그래도 정답은 "볼륨 크기 늘리기"야. 단, 이유가 달라.
gp3는 IOPS가 볼륨 크기와 완전히 분리(decoupled)돼 있어서 기본 3,000 IOPS를 제공하고, 크기와 무관하게 IOPS를 독립적으로 프로비저닝(최대 16,000)해. 그러니 크기를 늘려도 IOPS엔 애초에 영향이 없어.

정리하면 → gp2는 "8TB면 이미 16,000 캡에 도달해서" 안 되고, gp3는 "원래 크기와 IOPS가 무관해서" 안 돼. gp3에서의 올바른 해결책은 그냥 IOPS 값을 더 높게 프로비저닝하는 거야.
🎯 시험 팁 gp2에서 "IOPS 부족 + 이미 큰 볼륨" → 크기 증설은 함정. 정답 액션은 gp3/io 마이그레이션. gp2는 크기=IOPS(단 16,000 캡), gp3는 크기와 IOPS 무관(별도 프로비저닝) 공식을 꼭 기억.
7
여러 AZ에 분산된 EC2 집합이 대규모 데이터셋을 처리 중. 모든 EC2에서 NFS 드라이브로 동일 데이터에 접근하게 하려면?
▶
정답
🔀 Amazon EFS
여러 AZ에 걸친 여러 EC2가 동일 데이터를 NFS로 공유해야 하는 정확히 그 시나리오가 EFS야. EBS는 단일 AZ·기본 1대 연결이라 부적합하고, Instance Store는 임시 로컬이라 공유가 안 돼.
🎯 시험 팁 "여러 AZ" + "공유" + "NFS" 3개 키워드가 뜨면 반사적으로 EFS. (Linux 전용인 점도 같이 기억)
8
EC2 앱용 고성능 로컬 캐시가 필요. 인스턴스 종료 시 캐시가 사라져도 무방. 어떤 스토리지를 권장?
▶
정답
⚡ EC2 Instance Store
"고성능" + "로컬" + "사라져도 무방" = Instance Store의 교과서 정의야. 호스트에 물리적으로 붙어 있어 초고성능이고, 종료 시 데이터가 사라지는 임시 특성이 캐시/버퍼 용도와 완벽히 맞아. EBS를 쓰면 영구성 비용만 낭비돼.
🎯 시험 팁 "캐시/버퍼/스크래치 + 데이터 손실 OK + 고성능" → Instance Store. "손실되면 안 됨"이 붙으면 EBS로 바뀜.
9
기본 저장소에 310,000 IOPS를 요구하는 고성능 데이터베이스. 무엇을 추천?
▶
정답
⚡ EC2 Instance Store
EBS의 최상위 유형인 io2 Block Express도 최대 256,000 IOPS라, 단일 EBS 볼륨으로는 310,000 IOPS를 낼 수 없어. 이 수준의 IOPS는 물리적으로 직결된 Instance Store만 감당해(수십만~수백만 IOPS 가능). 단, 임시 저장소라 복제·백업은 애플리케이션 레벨에서 직접 책임져야 해.
🎯 시험 팁 IOPS 요구가 256,000을 초과하면 EBS로는 불가 → Instance Store. "수십만 IOPS 이상"이라는 숫자가 곧 힌트.
Cheat Sheet

빠른 요약 & 시험 TIP

시험장 직전에 훑어볼 압축 카드 + 키워드→정답 규칙.

📍 EBS 범위
단일 AZ에 고정. 다른 AZ로 옮기려면 스냅샷 → 복원.
🗑 종료 시 삭제
루트=true(삭제), 추가 EBS=false(보존)가 기본값.
💽 부팅 볼륨
SSD(gp2·gp3·io1·io2)만 가능. HDD(st1·sc1) 불가.
⚠️ gp2 IOPS 캡
3 IOPS/GB, 최대 16,000. ≈5.3TB부터 이미 캡.
🔗 Multi-Attach
io1/io2 전용, 같은 AZ, 최대 16 인스턴스.
🌎 AMI 범위
리전 단위. 타 리전은 Copy AMI 필요.
⚡ Instance Store
초고성능·임시. 종료 시 소실. 캐시/버퍼용.
🔀 EFS
NFS·다중 AZ·다중 EC2 공유. Linux 전용.
🎯 키워드 → 정답 매칭
  • 다른 AZ 인스턴스에 EBS 연결 안 됨 → 정상 동작(AZ 고정), 스냅샷으로 이동
  • 종료 시 루트 데이터 유지 → DeleteOnTermination = false
  • 타 리전에서 EC2 시작 → AMI Copy 후 사용
  • 부팅 볼륨 가능 유형 → gp2·gp3·io1·io2 (SSD)
  • gp2 IOPS 부족 + 이미 대용량 → gp3/io로 마이그레이션(크기 증설 ✘)
  • 여러 AZ · 공유 · NFS → EFS
  • 고성능 캐시 · 손실 OK → Instance Store
  • 256,000 IOPS 초과 요구 → Instance Store (EBS 불가)
  • 단일 볼륨 여러 EC2 동시 연결 → Multi-Attach(io1/io2)
