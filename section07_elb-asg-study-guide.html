Foundation

확장성(Scalability) — 시험 첫 문장

ELB/ASG 챕터의 모든 문제는 결국 "이건 수직이야 수평이야?"에서 출발합니다. 두 단어만 확실히 구분하면 됩니다.

⬆️ VerticalScale Up / Down

수직 확장성 (Vertical Scalability)

"노트북 RAM 8GB → 32GB로 갈아끼우기"

  • › 인스턴스 크기를 키움 (t2.micro → t2.large)
  • › r4.large → r4.4xlarge = 수직 확장
  • › 주로 분산이 어려운 시스템: RDS, ElastiCache
  • › 하드웨어 한계가 존재 → 무한히 못 키움
➡️ HorizontalScale Out / In

수평 확장성 (Horizontal Scalability)

"노트북 1대 → 노트북 5대로 늘리기"

  • › 인스턴스 개수를 늘림/줄임
  • › = Elasticity(탄력성), 분산 시스템의 기본
  • › ASG + ELB가 바로 이것
  • › 클라우드에서 사실상 무한 확장 가능
🛡️ HA≠ Scalability

고가용성 (High Availability)

"한 건물이 정전돼도 다른 건물에서 계속 서비스"

  • › 최소 2개 이상의 AZ에 인스턴스 배치
  • › 목표는 성능이 아니라 장애 생존
  • › 수평확장과 세트로 자주 등장하지만 다른 개념
🧠
프론트엔드 비유: 수직확장 = 내 맥북을 M1 → M3 Max로 교체. 수평확장 = Vercel이 트래픽 몰릴 때 서버리스 함수 인스턴스를 100개로 늘리는 것. 문제에 인스턴스 "타입/크기"가 바뀌면 수직, "개수"가 바뀌면 수평.
Load Balancer Types

ALB vs NLB vs GWLB — 진짜 차이

"NLB는 TCP/UDP라며? 근데 HTTP도 처리하잖아?" — 이 혼란이 정확히 시험에서 노리는 지점입니다. 아래에서 해소합니다.

⚠️
결정적 오해 정리: HTTP/HTTPS는 TCP 위에서 돌아가는 프로토콜입니다. 그래서 NLB(Layer 4)로도 HTTP 트래픽을 흘려보낼 수 있습니다. 차이는 "통과시키느냐"가 아니라 "내용을 읽고 판단할 수 있느냐"입니다.

· ALB(L7): HTTP를 열어서 읽음 → URL 경로, 헤더, 쿠키, 호스트명 보고 라우팅 결정 가능
· NLB(L4): IP·포트만 보고 봉투째 전달 → 안의 HTTP 내용은 못 봄. 대신 압도적으로 빠름
📦 OSI 계층으로 보는 ELB

NLB는 어떻게 대상을 고르나 — Flow Hash

ALB에는 "규칙(Rule)"이 있는데, NLB에는 규칙이 없습니다. 그럼 대상 5대 중 누구에게 보낼지 어떻게 정할까요?

1층리스너

포트로만 갈린다

"NLB의 유일한 분기 조건 = 들어온 포트"

  • › TCP 80 → 대상 그룹 A
  • › TCP 443 → 대상 그룹 B
  • › UDP 5000 → 대상 그룹 C
  • › 리스너 1개 = 대상 그룹 1개. 경로/헤더 조건 불가
2층Flow Hash

대상 그룹 안에서 누구?

"연결의 지문을 해시 돌려서 한 명 지목"

  • › 해시 입력: 프로토콜 · 출발지 IP · 출발지 포트 · 목적지 IP · 목적지 포트 · TCP 시퀀스 번호
  • › 새 TCP 연결이 맺어질 때 딱 1회 계산
  • › 한 번 정해지면 그 연결이 끊길 때까지 같은 대상 유지
  • › 요청 단위가 아니라 연결(flow) 단위
⚡ 성능

NLB가 빠른 진짜 이유

"TCP가 가벼워서가 아니라, 안 하는 일이 많아서"

  • › ① 연결을 끊고 다시 맺지 않음 (ALB는 완전 프록시라 핸드셰이크 2번)
  • › ② HTTP를 파싱하지 않음 (헤더·경로·쿠키 안 읽음)
  • › ③ 규칙 매칭이 없음 — 해시 한 번이면 끝
  • › ④ AWS의 Hyperplane 네트워크 인프라 위에서 동작
⚠️
흔한 오해: "NLB는 TCP라서 HTTP보다 가볍다" — 틀렸습니다. HTTP는 TCP 위에서 도는 프로토콜이라 ALB도 밑바닥은 똑같이 TCP를 씁니다. 차이는 프로토콜의 무게가 아니라 ALB가 추가로 하는 일(연결 종료 + 파싱 + 규칙 매칭)에 있습니다.

🧠
프론트엔드 비유:
· ALB = 라우터 — react-router가 pathname을 읽고 컴포넌트를 고르듯, 내용을 열어보고 분기
· NLB = 파이프 — socket.pipe(target). 내용을 안 보고 그대로 흘려보냄

그래서 순수 TCP/UDP 서버(게임 서버, MQTT)는 ALB로 아예 다룰 수 없고, 경로별 라우팅은 NLB로 절대 못 합니다.
💡
예외: NLB에 TLS 리스너를 걸면 복호화를 위해 NLB도 연결을 종료합니다. 순수 TCP 패스스루일 때가 가장 빠릅니다.
또한 NLB의 Sticky Session은 쿠키가 아니라 출발지 IP 기반(Source IP Affinity)입니다 — 쿠키는 HTTP 개념이라 L4에서는 쓸 수 없기 때문입니다.
The Big Confusion

대상 그룹(Target Group)은 도대체 뭔가

"로드밸런서를 지웠는데 대상 그룹은 남아있고, ASG는 잘 돌아간다는데... 그럼 로드밸런서는 필요없는 거 아냐?" — 이 질문 하나만 풀면 이 챕터가 다 풀립니다.

한 문장 정의

🎯
대상 그룹 = "트래픽을 받을 대상들의 명단 + 그 대상들의 건강검진 규칙(Health Check)". 그 이상도 이하도 아닙니다. 대상 그룹 자체는 트래픽을 전혀 보내지 않습니다.

그래서 각자 무슨 일을 하나

구성 요소역할 (딱 이것만)없으면 생기는 일
로드 밸런서트래픽을 실제로 받아서 분배. 외부에 노출되는 DNS 주소의 주인.사용자가 접속할 입구 자체가 없음

규칙 (Rule)조건(경로·호스트명 등)에 따라 어느 대상 그룹으로 보낼지 결정라우팅 분기 불가
대상 그룹명단 + 헬스체크 규칙만 보관. 능동적으로 아무 것도 안 함.LB가 "누구한테 보내야 하지?"를 모름
ASG인스턴스 개수 유지·증감. 새로 만든 인스턴스를 대상 그룹에 자동 등록.수동으로 인스턴스 늘리고 등록해야 함
⚠️
의문에 대한 답: 로드밸런서를 지우면 로드밸런싱은 더 이상 일어나지 않습니다. 대상 그룹이 남아있는 건 그냥 "명단 종이 한 장"이 서랍에 남아있는 것과 같습니다. ASG는 로드밸런싱을 하는 기능이 아닙니다. ASG는 오직 "인스턴스를 몇 대 유지할지"만 담당합니다. AWS 콘솔에서 ASG가 "정상"으로 보이는 건 인스턴스 개수를 잘 유지하고 있다는 뜻이지, 트래픽이 분산되고 있다는 뜻이 아닙니다.
🎭 역할 분리

ASG ≠ 로드밸런싱

"ASG는 인사팀, LB는 안내데스크"

  • › ASG: 직원(EC2)이 몇 명인지 관리. 아프면 해고하고 새로 채용.
  • › LB: 손님(요청)을 누구에게 안내할지 결정.
  • › 대상 그룹: 안내데스크가 보는 재직자 명단.
  • › 인사팀이 있어도 안내데스크가 없으면 손님은 못 들어옴.
🧩 왜 분리했나

대상 그룹이 별도로 존재하는 이유

"React의 state를 컴포넌트 밖으로 뺀 것과 같음"

  • › 하나의 대상 그룹을 여러 LB가 공유 가능
  • › 하나의 LB가 여러 대상 그룹으로 분기 (경로별 라우팅)
  • › 헬스체크 설정을 재사용 가능
  • › ASG는 LB를 몰라도 됨 — 대상 그룹에만 등록하면 끝
🔄 흐름

ASG + 대상 그룹 연동

  • › ASG에 대상 그룹을 연결(attach)해 둠
  • › ASG가 새 인스턴스 시작 → 자동으로 대상 그룹에 등록
  • › ASG가 인스턴스 종료 → 자동으로 대상 그룹에서 제거
  • › 그래서 인스턴스를 지워도 대상 그룹은 그대로 남음 (빈 명단)

"인스턴스가 충돌한다"는 게 무슨 뜻인가

🧠
충돌(crash) = 인스턴스는 켜져 있는데 앱이 죽은 상태. EC2 하드웨어가 폭발한 게 아니라, 그 안에서 돌던 Node 프로세스가 OOM으로 죽거나, 무한루프에 빠져 응답을 못 하는 상황을 말합니다.

EC2 상태 검사(Status Check)는 "기계가 살아있나"만 보므로 여전히 정상으로 나옵니다. 반면 ELB Health Check는 GET /health를 실제로 쏴보고 200이 안 오면 비정상으로 판단해 그 인스턴스로 트래픽을 안 보냅니다. 그래서 사용자가 "가끔 안 돼요"라고 하면 답은 항상 ELB Health Check입니다.
Core Features

Sticky Session · X-Forwarded-For · Cross-Zone

ELB 문제의 절반은 이 세 기능 중 하나를 묻습니다.

🍪 Sticky Session세션 고정

Sticky Session (Session Affinity)

"같은 손님은 항상 같은 웨이터에게"

  • › 같은 클라이언트를 항상 같은 인스턴스로 보냄
  • › 쿠키로 구현. ALB / NLB 모두 지원(CLB도)
  • › 안 켜면 → 페이지 이동마다 다른 서버 → 재로그인 요구
  • › 단점: 특정 인스턴스에 부하가 쏠릴 수 있음
🍪 쿠키 종류

ALB 쿠키 3종

  • › AWSALB — ALB가 생성 (예약어)
  • › AWSALBAPP — 애플리케이션 기반 (예약어)
  • › AWSALBTG — 대상 그룹용 (예약어)
  • › ⛔ 커스텀 쿠키 이름으로 위 3개 사용 불가
  • › ✅ 그 외 아무 이름 (MySessionCookie 등) 사용 가능
📮 X-Forwarded-*

클라이언트 IP 되찾기

"우체국을 거치면 발신인이 우체국으로 찍힌다"

  • › ALB는 연결을 종료하고 새로 맺음 → 앱에는 ALB의 프라이빗 IP만 보임
  • › X-Forwarded-For → 원본 클라이언트 IP
  • › X-Forwarded-Proto → 원본 프로토콜(http/https)
  • › X-Forwarded-Port → 원본 포트
  • › NLB는 L4라 클라이언트 IP가 그대로 보존됨 (헤더 불필요)
🌍 Cross-Zone

Cross-Zone Load Balancing

"AZ 단위가 아니라 인스턴스 단위로 공평하게"

  • › OFF: 각 AZ의 LB 노드가 자기 AZ 인스턴스에만 분배 → AZ간 인스턴스 수가 다르면 부하 불균형
  • › ON: 모든 LB 노드가 전체 AZ의 모든 인스턴스에 균등 분배
  • › ALB: 기본 ON, 끌 수 없음, AZ간 요금 무료
  • › NLB / GWLB: 기본 OFF, 켜면 AZ간 데이터 전송 요금 발생
💡
Cross-Zone 계산 예시: AZ-a에 2대, AZ-b에 8대, 총 트래픽 100%.
· OFF → AZ-a가 50%를 2대가 나눠먹음(각 25%), AZ-b는 50%를 8대가 나눠먹음(각 6.25%). AZ-a 인스턴스가 CPU 폭발.
· ON → 10대가 각각 10%씩. 균등.
시험에서 "AZ간에는 트래픽이 균등한데 특정 AZ 인스턴스만 CPU가 높다" → 답은 무조건 Cross-Zone Load Balancing 활성화.

  • › 클라이언트가 TLS 핸드셰이크 초기에 원하는 호스트명을 알려줌
  • › LB가 그걸 보고 맞는 인증서를 골라서 제시
  • › ✅ 지원: ALB, NLB, CloudFront
  • › ❌ 미지원: CLB (Classic LB) — 인증서 1개만
🛠️ 설정 순서

ALB에 HTTPS 붙이기

  • › ① ACM(AWS Certificate Manager)에서 인증서 발급/가져오기
  • › ② ALB에 HTTPS(443) 리스너 생성
  • › ③ 리스너에 기본 인증서 1개 지정
  • › ④ 인증서 목록에 추가 인증서들 등록 → SNI가 자동으로 매칭
  • › ⑤ 보안 정책(TLS 버전/암호 스위트) 선택
🔐 SG 규칙

보안 그룹은 이렇게 짠다

  • › LB의 SG: 인바운드 0.0.0.0/0 에서 80/443 허용

  • › → EC2는 오직 LB를 통해서만 접근 가능 (직접 접근 차단)
  • › 이게 시험에 나오는 표준 패턴

Auto Scaling Group (ASG)

ASG의 정체성: "인스턴스 개수를 목표치에 맞추는 기계". 그 이상의 일은 하지 않습니다.

3개의 숫자

속성의미시험 포인트
Minimum (최소)이 아래로는 절대 줄지 않음축소(Scale In)의 하한선
Desired (목표)지금 유지하려는 개수스케일링은 이 값을 바꾸는 것
Maximum (최대)이 위로는 절대 늘지 않음Desired == Max면 CPU 100%여도 아무 일 없음

헬스체크 & 쿨다운

개념설명시험 함정
EC2 Status CheckASG 기본 헬스체크. 하드웨어/OS 수준만 확인.앱이 죽어도 "정상"으로 봄
ELB Health CheckASG에 활성화하면 ELB 판정을 그대로 따름. 앱 응답을 실제로 확인.비정상 → ASG가 인스턴스 종료 후 새로 시작
Cooldown (기본 300초)스케일링 활동 후 5분간 추가 스케일링을 하지 않음."트래픽 계속 느는데 5분 뒤에야 늘어남" → Cooldown
Instance Warm-up새 인스턴스를 N초 동안 지표 계산에서 제외. 부팅 중 CPU 100%가 평균을 오염시키는 걸 방지."부팅 중 CPU 때문에 과잉 확장" → Warm-up

Cooldown vs Warm-up — 가장 잘 틀리는 구분

🧠
· Cooldown = "방금 스케일링했으니 좀 쉬어"
  → 스케일링 활동 자체를 일시 정지. 기본 300초. 주로 Simple / Step Scaling용.
  → 증상: "트래픽이 계속 느는데 5분 뒤에야 인스턴스가 추가됨"

· Warm-up = "새로 온 애 지표는 아직 믿지 마"
  → 지표 계산에서 신규 인스턴스를 제외. 주로 Target Tracking용.
  → 증상: "새 인스턴스가 부팅 중이라 CPU 100%인데, 이게 평균에 잡혀서 또 확장됨"

핵심 차이: Cooldown은 시간을 막고, Warm-up은 데이터를 거릅니다.
⚠️
ASG의 "복구"는 치료가 아니라 교체입니다. 비정상 인스턴스를 고치는 게 아니라 종료(terminate)하고 새 인스턴스를 시작합니다. Desired 개수를 맞추는 게 목적이기 때문입니다.

🩺 unhealthy 디버깅 체크리스트

"SSH로 들어가서 확인하면 웹서버는 잘 도는데 대상 그룹에선 unhealthy" — 원인은 거의 항상 이 둘입니다.

의심 지점확인 방법해결
① 보안 그룹EC2 SG의 인바운드가 ALB의 SG를 허용하는가?인바운드 소스를 LB의 보안 그룹 ID로 지정
② 헬스체크 경로ALB가 /health를 때리는데 앱이 404를 반환하지 않는가?경로를 실제 존재하는 곳으로 변경, 또는 성공 코드 조정
③ 포트헬스체크 포트와 앱 리스닝 포트가 일치하는가?대상 그룹의 포트 설정 확인
④ 임계값앱 기동이 느려 헬스체크 타임아웃에 걸리지 않는가?Interval / Timeout / Unhealthy threshold 조정
💡
왜 헷갈리나: SSH로 들어가 curl localhost:80 하면 잘 됩니다. 하지만 ALB는 외부에서 /health를 물어보고 있어요. 서버는 살아있지만 ALB가 던진 질문에 답을 못 한 것입니다. Cross-Zone, Sticky Session, SNI는 헬스체크와 전혀 무관합니다 — 오답 선택지로 자주 나옵니다.
Exam Practice

시험 예상 문제 25

클릭하면 정답과 해설이 펼쳐집니다. 오답 선택지가 왜 틀렸는지까지 확인하세요.

1
EC2 인스턴스를 r4.large에서 r4.4xlarge로 확장하는 것을 무엇이라고 합니까?
▶
정답
⬆️ 수직 확장성 (Vertical Scalability)
인스턴스의 크기(타입)가 커졌습니다. 개수는 그대로 1대입니다. 크기를 키우는 것은 Scale Up = 수직 확장.
수평 확장은 인스턴스 개수가 늘어나야 합니다.
🎯 시험 팁 인스턴스 타입/사이즈가 바뀌면 → 수직. RDS, ElastiCache처럼 분산이 어려운 서비스는 수직 확장이 기본.
2
EC2 인스턴스 수를 확장 및 축소하는 Auto Scaling Group에서 애플리케이션을 실행하는 것을 무엇이라고 합니까?
▶
정답
➡️ 수평 확장성 (Horizontal Scalability)
인스턴스의 개수가 늘고 줄어듭니다. Scale Out / Scale In이며, 이를 클라우드에서는 Elasticity(탄력성)라고도 부릅니다.
🎯 시험 팁 ASG라는 단어가 보이면 → 수평 확장. 자동으로 튀어나와야 하는 반사신경.
3
ELB는 무엇을 제공합니까?
▶
정답
🌐 정적 DNS 이름 (Static DNS Name)
ELB는 변하지 않는 DNS 엔드포인트를 제공합니다. 내부 노드의 IP는 수시로 바뀌기 때문에 정적 IP는 제공하지 않습니다.
단, NLB만은 예외로 AZ당 정적 IP를 가지며 Elastic IP도 붙일 수 있습니다.
🎯 시험 팁 ELB(일반) → static DNS / NLB → static IP. 이 두 줄이 헷갈리면 문제 3번과 11번을 동시에 틀립니다.
4
ELB 뒤에 10개의 EC2가 있는 웹사이트에서 사용자가 페이지를 이동할 때마다 재인증을 요구받습니다. 개발 환경(EC2 1대)에서는 정상입니다. 원인은?
▶
정답
🍪 Sticky Session(고정 세션)이 비활성화되어 있음
각 EC2가 자기 메모리에 세션을 들고 있는데, 요청마다 다른 EC2로 분산되니 세션을 못 찾아 재로그인을 요구합니다. EC2가 1대일 땐 항상 같은 서버라 문제가 안 보였던 것.
해결: 대상 그룹에서 Stickiness를 활성화 → 같은 클라이언트를 항상 같은 인스턴스로 보냄.
🎯 시험 팁 "로그인이 자꾸 풀린다 / 장바구니가 사라진다" → Sticky Session. (근본 해결은 ElastiCache나 DynamoDB에 세션 외부 저장)
5
ALB 뒤의 웹사이트가 클라이언트 IP 대신 ALB의 프라이빗 IPv4만 봅니다. 실제 클라이언트 IP를 얻으려면?
▶
정답
📮 X-Forwarded-For 헤더를 읽는다
ALB는 클라이언트와의 연결을 끊고 EC2와 새 연결을 맺습니다. 그래서 EC2 입장에선 발신자가 ALB입니다. ALB는 원본 정보를 헤더에 담아 전달합니다:
· X-Forwarded-For → 원본 클라이언트 IP
· X-Forwarded-Proto → 원본 프로토콜
· X-Forwarded-Port → 원본 포트
🎯 시험 팁 "ALB인데 클라이언트 IP를 알고 싶다" → X-Forwarded-For. NLB는 L4라서 클라이언트 IP가 그대로 보존되므로 이 헤더가 필요 없습니다.
6
일부 EC2 인스턴스가 때때로 충돌해 사용자가 접속하지 못합니다. 사용자를 보호하려면?
▶
정답
🩺 ELB Health Check (상태 확인) 활성화
"인스턴스가 충돌한다"는 하드웨어 고장이 아니라 인스턴스는 켜져 있는데 그 안의 애플리케이션이 죽거나 응답 불능이 된 상태입니다. (Node 프로세스 OOM, 데드락 등)

ELB Health Check는 지정한 경로(예: /health)로 주기적으로 요청을 보내 응답 코드를 확인합니다. 비정상 판정된 인스턴스는 대상 그룹에서 트래픽을 받지 않게 됩니다. 사용자는 죽은 인스턴스로 라우팅되지 않습니다.
🎯 시험 팁 "가끔 안 된다 / 간헐적 오류" → Health Check. EC2 Status Check는 기계만 보므로 앱 장애를 못 잡습니다.
7
초당 수백만 건의 요청을 처리할 고성능·저지연 애플리케이션. 어떤 ELB를 선택합니까?
▶
정답
⚡ Network Load Balancer (NLB)
NLB는 Layer 4에서 패킷을 열어보지 않고 그대로 전달하므로 오버헤드가 거의 없습니다. 초당 수백만 요청, 극한의 저지연이 필요할 때 선택합니다.
ALB는 HTTP 내용을 파싱하므로 상대적으로 느립니다. GWLB는 성능용이 아니라 트래픽 검사용입니다.
🎯 시험 팁 "초당 수백만 / 초저지연 / 극한 성능" → NLB 반사적으로.
8
Network Load Balancer가 지원하는 프로토콜은?
▶
정답
🔌 TCP (그리고 UDP, TLS)
NLB는 Layer 4 = 전송 계층이므로 TCP / UDP / TLS를 지원합니다.

⚠️ "그럼 HTTP는?" — HTTP/HTTPS는 TCP 위에서 동작하므로 NLB로 흘려보낼 수는 있습니다. 하지만 NLB는 그 내용을 이해하지 못합니다. 경로(/api)나 헤더로 라우팅하는 건 불가능합니다. 그래서 시험에서 "NLB의 프로토콜"을 물으면 답은 TCP/UDP/TLS이지 HTTP가 아닙니다.
🎯 시험 팁 ALB = HTTP/HTTPS/gRPC/WebSocket, NLB = TCP/UDP/TLS, GWLB = IP 패킷(GENEVE 6081).
9
ALB가 조건에 따라 다른 대상 그룹으로 라우팅할 수 있는 조건이 아닌 것은?
▶
정답
🚫 클라이언트의 지리적 위치 (Geolocation)
ALB가 라우팅 조건으로 쓸 수 있는 것 (✅):
· 호스트 헤더 (api.example.com)
· 경로 (/users, /posts)
· 쿼리 문자열 (?platform=mobile)
· HTTP 헤더 / HTTP 메서드
· 소스 IP CIDR

못 쓰는 것 (❌): 지리적 위치(Geolocation) — 이건 Route 53의 라우팅 정책이거나 CloudFront의 기능입니다.
🎯 시험 팁 Geolocation / Geoproximity → Route 53. ALB의 조건은 "HTTP 요청 안에 있는 정보 + 소스 IP"까지만.
10
ALB의 대상 그룹에 등록할 수 있는 대상이 아닌 것은?
▶
정답
🚫 Network Load Balancer
ALB 대상 그룹에 등록 가능 (✅):
· EC2 인스턴스 (ASG로 관리 가능)
· ECS 태스크
· Lambda 함수 (HTTP 요청 → JSON 이벤트로 변환)
· 프라이빗 IP 주소 (온프레미스 서버 포함)

불가 (❌): NLB를 ALB의 대상으로 등록. 방향이 반대입니다 — ALB를 NLB의 대상으로 등록하는 것은 가능합니다 (NLB로 고정 IP를 얻고 뒤에 ALB의 L7 기능을 쓰는 조합).
🎯 시험 팁 ALB → NLB의 대상 (O) / NLB → ALB의 대상 (X). 방향을 기억하세요.
11
최종 사용자에게 고정 static IP를 노출해 규제 기관 승인 방화벽 규칙을 만들려 합니다. 어떤 ELB?
▶
정답
📌 Network Load Balancer (NLB)
NLB는 활성화된 AZ마다 고정 IP 1개를 가지며, Elastic IP를 직접 할당할 수도 있습니다. 방화벽 화이트리스트에 등록해야 하는 상황에 유일한 선택지입니다.
ALB는 정적 DNS만 제공하고 내부 IP는 수시로 바뀝니다.
🎯 시험 팁 "고정 IP / 화이트리스트 / 방화벽 규칙" → NLB. "고정 DNS" → ALB 포함 모든 ELB.
12
ALB에서 사용자 지정 애플리케이션 기반 쿠키를 만들려 합니다. 쿠키 이름으로 사용할 수 있는 것은?
▶
정답
🍪 예약어가 아닌 아무 이름 (예: MySessionCookie, user-session)
ALB가 예약해 둔 쿠키 이름 3개는 커스텀 쿠키로 쓸 수 없습니다:
· AWSALB — ALB 자체 생성 쿠키
· AWSALBAPP — 애플리케이션 기반 쿠키용
· AWSALBTG — 대상 그룹용

이 세 개를 제외한 이름이면 무엇이든 가능합니다. 선택지에서 AWS로 시작하는 것들을 지우면 답이 나옵니다.
🎯 시험 팁 예약어 3개는 앞글자만 외우세요: AWSALB / AWSALBAPP / AWSALBTG = "ALB, ALB앱, ALB타겟그룹".
13
NLB가 us-east-1b에 EC2 2개, us-east-1e에 EC2 5개로 분산 중. 트래픽은 두 AZ에 균등한데 1b의 CPU가 훨씬 높습니다. 해결책은?
▶
정답
🌍 Cross-Zone Load Balancing 활성화
Cross-Zone이 꺼져 있으면 트래픽이 AZ 단위로 50:50 분배됩니다.
· us-east-1b: 50% ÷ 2대 = 인스턴스당 25%
· us-east-1e: 50% ÷ 5대 = 인스턴스당 10%
→ 1b의 CPU가 2.5배 높은 게 당연합니다.

Cross-Zone을 켜면 전체 7대에 균등 분배되어 인스턴스당 약 14.3%가 됩니다.

NLB는 Cross-Zone이 기본 비활성화이므로 이 문제가 발생합니다. (ALB는 기본 활성화라 이 문제가 안 생깁니다.)
🎯 시험 팁 "AZ간 트래픽은 균등한데 특정 AZ의 CPU만 높다" → Cross-Zone LB 활성화. 그리고 NLB/GWLB는 기본 OFF (켜면 AZ간 데이터 요금 발생), ALB는 기본 ON (무료).
14
ALB와 NLB의 어떤 기능으로 하나의 리스너에 여러 SSL 인증서를 로드할 수 있습니까?
▶
정답
📜 SNI (Server Name Indication)
SNI는 클라이언트가 TLS 핸드셰이크 초기에 접속하려는 호스트명을 평문으로 알려주는 확장 기능입니다. LB는 그 호스트명을 보고 맞는 인증서를 골라서 제시합니다.

✅ 지원: ALB, NLB, CloudFront
❌ 미지원: Classic Load Balancer (인증서 1개만 가능 → LB를 여러 개 만들어야 함)
🎯 시험 팁 "리스너 1개 + 인증서 여러 개" → SNI. "CLB로 여러 도메인 HTTPS?" → 불가능, LB를 도메인마다 따로.
15
ALB가 users.example.com, api.external.example.com, checkout.example.com 3개 호스트명으로 3개 대상 그룹에 라우팅합니다. 각각 HTTPS를 구성하려면?
▶
정답
🔒 HTTPS 리스너 1개를 만들고, SNI를 이용해 인증서 3개를 등록
ALB의 HTTPS(443) 리스너에는 기본 인증서 1개 + 추가 인증서 여러 개를 등록할 수 있습니다. 클라이언트가 SNI로 호스트명을 알려주면 ALB가 자동으로 맞는 인증서를 선택합니다.

리스너를 3개 만든다 → 리스너는 포트 기준이므로 443 포트에 3개를 만들 수 없습니다.
ALB를 3개 만든다 → 동작은 하지만 비용/관리 측면에서 오답. (CLB라면 이 방법밖에 없음)

※ api.external.example.com처럼 서브도메인이 여러 단계여도 문제없습니다. 하나의 와일드카드 인증서(*.example.com)로는 api.external.example.com을 커버 못 한다는 점만 주의(와일드카드는 한 단계만).
🎯 시험 팁 "여러 도메인 + HTTPS + ALB" → 리스너 1개 + 다중 인증서 + SNI. 인증서는 ACM에서 발급.
16
ASG의 Desired=3, Max=3. CPU 60% 도달 시 확장하는 CloudWatch Alarm이 있습니다. 트래픽 급증으로 CPU가 80%가 되면?
▶
정답
😐 아무 일도 일어나지 않는다
알람은 정상적으로 울리고 스케일링 정책도 트리거됩니다. 하지만 Desired(3) == Maximum(3)이므로 ASG는 더 이상 인스턴스를 추가할 수 없습니다.

Maximum은 절대 넘을 수 없는 천장입니다. 확장하려면 Max 값을 먼저 올려야 합니다.
"인스턴스가 1개 추가된다" → Max에 막힘.
"알람이 울리지 않는다" → 알람은 울립니다. 실행이 안 될 뿐.
🎯 시험 팁 ASG 문제는 항상 Min/Desired/Max 숫자부터 확인. Desired가 Max에 닿아 있으면 답은 "아무 일 없음".
17
ALB 앞에 ASG가 있고, ASG가 ALB Health Check를 사용하도록 설정했습니다. EC2 하나가 비정상으로 보고되면 어떻게 됩니까?
▶
정답
♻️ ASG가 해당 인스턴스를 종료하고 새 인스턴스를 시작한다
ASG의 헬스체크 유형을 ELB로 설정하면, ASG는 ALB의 판정을 자기 판정으로 받아들입니다. 비정상 인스턴스는:
① 대상 그룹에서 등록 해제(deregister) → 트래픽 차단
② ASG가 종료(terminate)
③ Desired 개수를 맞추기 위해 새 인스턴스 시작

"인스턴스를 재부팅한다" → ASG는 고치지 않습니다. 교체합니다.
"아무 일도 안 일어난다" → ELB 헬스체크를 켰으므로 반응합니다.
🎯 시험 팁 ASG의 복구 = 치료가 아니라 교체(terminate & replace). 기본 헬스체크는 EC2 Status Check(하드웨어만)라서 앱 장애를 못 잡습니다.
18
애플리케이션이 데이터베이스에 보내는 분당 요청 수를 기준으로 ASG를 스케일링하고 싶습니다. 어떻게 해야 합니까?
▶
정답
📊 CloudWatch 사용자 지정 지표(Custom Metric)를 게시하고, 그 지표에 CloudWatch Alarm을 만들어 ASG 스케일링 정책에 연결
"분당 DB 요청 수"는 AWS가 기본 제공하는 지표가 아닙니다. 따라서:
① 애플리케이션에서 PutMetricData API로 CloudWatch에 커스텀 지표 게시
② 그 지표를 기준으로 CloudWatch Alarm 생성
③ 알람을 ASG의 Simple/Step Scaling Policy에 연결

"CPU 사용률로 스케일링한다" → 요구사항과 다릅니다.
"Scheduled Scaling" → 시간 기준이지 지표 기준이 아닙니다.
🎯 시험 팁 "AWS 기본 지표가 아닌 무언가로 스케일링" → Custom Metric + CloudWatch Alarm. 게시 API는 PutMetricData.
19
ALB + ASG 환경. EC2 인스턴스당 평균 연결 수를 약 1000개로 유지하는 스케일링 정책을 정의하려 합니다. 어떤 정책?
▶
정답
🎯 대상 추적 스케일링 정책 (Target Tracking Scaling Policy)
"~을 ~값으로 유지"라는 문장은 Target Tracking의 정의 그 자체입니다. 목표값(1000)만 지정하면 AWS가 필요한 CloudWatch Alarm을 자동으로 생성·관리합니다.
대표 지표: ALBRequestCountPerTarget, ASGAverageCPUUtilization

Step Scaling → "임계값 초과 시 N대 추가" 형태. 유지가 아닌 반응.
Scheduled Scaling → 시간 기반.
Simple Scaling → 알람 하나에 액션 하나. 목표 유지 개념 없음.
🎯 시험 팁 "평균 ~을 ~로 유지(maintain / average around)" → Target Tracking. "~을 넘으면 몇 대 추가" → Step/Simple.
20
ASG가 확장되어 새 EC2를 시작했습니다. 트래픽은 계속 증가하는데 ASG는 다음 인스턴스를 즉시 시작하지 않고 5분 후에 시작합니다. 원인은?
▶
정답
⏳ Cooldown Period (기본값 300초 = 5분)
스케일링 활동이 발생하면 ASG는 기본 300초의 쿨다운에 들어갑니다. 이 동안에는 추가 스케일링 활동을 하지 않습니다.

이유: 방금 시작한 인스턴스가 부팅·앱 기동을 마치고 실제로 트래픽을 받아 지표에 반영되기까지 시간이 걸립니다. 그 전에 계속 인스턴스를 늘리면 과잉 확장(over-scaling)이 발생합니다.

숫자 300초 / 5분이 문제에 나오면 거의 확정적으로 Cooldown입니다.
※ 관련 개념: Warm-up은 Target Tracking에서 새 인스턴스를 지표 계산에 포함하기까지 기다리는 시간입니다.
🎯 시험 팁 "5분 / 300초 지연 후에야 스케일링" → Cooldown Period. AMI를 미리 만들어 부팅 시간을 줄이면 쿨다운을 짧게 설정할 수 있습니다.
🔥
추가 5문제 — 특히 헷갈리기 쉬운 지점(UDP, unhealthy 진단, Target Tracking 양방향, Warm-up)만 골랐습니다.
21
게임 회사가 실시간 멀티플레이 서버를 배포합니다. 클라이언트는 UDP로 통신하며 초저지연이 필수이고, 서버는 부정행위 탐지를 위해 클라이언트의 실제 IP를 알아야 합니다. 어떤 구성이 적절합니까?
▶
정답
⚡ Network Load Balancer (NLB)
UDP라는 단어가 나온 순간 ALB는 즉시 탈락입니다. ALB는 HTTP / HTTPS / gRPC / WebSocket만 다룹니다.

NLB는 TCP/UDP/TLS를 지원하고, L4 패스스루라서 클라이언트 IP가 그대로 보존됩니다. X-Forwarded-For는 HTTP 헤더이므로 UDP에는 애초에 존재하지도 않습니다.
GWLB + 프록시 프로토콜 → GWLB는 트래픽 검사 장비 앞단용이지 게임 서버 분산용이 아닙니다.
🎯 시험 팁 UDP → NLB 확정. 그리고 "클라이언트 IP 보존"은 NLB의 기본 성질(설정 불필요), ALB는 X-Forwarded-For 필요.
22
ALB 뒤의 EC2에 SSH로 접속해 확인하니 웹 서버는 정상 동작합니다. 그런데 대상 그룹에서는 unhealthy로 표시됩니다. 가능성 높은 원인 두 가지는?
▶
정답
🩺 ① EC2 보안 그룹이 ALB의 SG를 허용하지 않음  +  ② 헬스체크 경로가 잘못됨
왜 SSH로는 잘 되는데 unhealthy인가? SSH로 들어가 curl localhost:80 하면 당연히 잘 됩니다. 하지만 ALB는 외부에서 지정된 경로(/health 등)를 물어보고 있어요.

· ① SG: EC2의 인바운드가 ALB의 보안 그룹을 허용하지 않으면 헬스체크 요청 자체가 도달하지 못합니다. 응답이 없으니 unhealthy.
· ② 경로: 헬스체크는 성공 코드(기본 200)를 기대합니다. /health 라우트가 없어 404가 오면 실패 판정.

Cross-Zone LB → 부하 분산 비율 문제. 헬스체크 무관.
Sticky Session → 세션 고정 기능. 헬스체크 무관.
SNI → 인증서 매칭 기능. 헬스체크 무관.
🎯 시험 팁 unhealthy → ① SG 인바운드 ② 헬스체크 경로/상태코드. 이 둘만 보면 됩니다. Cross-Zone·Sticky·SNI는 단골 오답 선택지.
23
ASG의 Min=2, Desired=4, Max=8. Target Tracking 정책은 "평균 CPU 40% 유지". 현재 CPU가 15%로 떨어졌습니다. 어떻게 됩니까?
▶
정답
📉 인스턴스가 2개(Minimum)까지 축소된다
Target Tracking은 양방향입니다. 목표(40%)보다 높으면 확장(Scale Out), 낮으면 축소(Scale In)합니다. CPU 15%면 인스턴스가 남아도는 상황이므로 줄이려 하고, Minimum(2)이 바닥이라 2개까지 내려갑니다.

"0개까지 축소" → Min이 2라 불가능.
"4개로 유지" → Desired는 스케일링의 결과값이지 고정값이 아닙니다.
"아무 일도 일어나지 않는다" → 이건 16번 문제(Desired == Max, 천장에 막힘)의 답입니다. 여기선 바닥이 아직 남아있으니 움직입니다.
🎯 시험 팁 Target Tracking = 에어컨. 더우면 세게, 추우면 약하게. 확장만 하는 게 아닙니다. 천장 = Max, 바닥 = Min. 문제에서 어느 쪽에 부딪히는지 확인하세요.
24
checkout.example.com(결제)과 www.example.com(일반)을 하나의 ALB로 서비스합니다. 결제 트래픽은 PCI 규정상 별도의 강화된 EC2 그룹에서 처리해야 합니다. 어떻게 구성합니까?
▶
정답
🔀 하나의 리스너에 호스트 헤더 조건 규칙 → 서로 다른 대상 그룹
정확히 ALB(L7)가 존재하는 이유입니다. HTTPS 리스너 하나에:
· 규칙 1: Host == checkout.example.com → 대상 그룹 [PCI-hardened]
· 규칙 2: Host == www.example.com → 대상 그룹 [general]
인증서는 SNI로 두 개 모두 등록하면 됩니다.

ALB 2개 + Route 53 가중치 → 동작은 하지만 "하나의 ALB로"라는 조건 위반 + 과도한 구성.
NLB 앞 + ALB 2개 → 불필요한 계층 추가.
대상 그룹 하나에 다 등록 + Sticky Session → PCI 격리가 무너집니다. Sticky Session은 세션 고정 기능이지 격리 수단이 아닙니다.
🎯 시험 팁 "호스트명/경로별로 다른 서버" → ALB 규칙 + 다중 대상 그룹. "규제 격리"가 나오면 대상 그룹을 반드시 분리.
25
ASG가 새 인스턴스를 시작한 직후, 그 인스턴스는 앱 부팅 중이라 CPU가 100%입니다. Target Tracking 정책이 이 값을 평균에 포함시켜 불필요하게 인스턴스를 더 시작합니다. 무엇을 설정해야 합니까?
▶
정답
🔥 Instance Warm-up 시간 설정
Warm-up은 "새 인스턴스는 N초 동안 지표 계산에서 제외하라"는 설정입니다. 부팅 중 튀는 CPU가 평균을 오염시켜 과잉 확장(over-scaling)을 유발하는 걸 막아줍니다.

Cooldown을 600초로 늘린다 → Cooldown은 주로 Simple/Step Scaling용이고, Target Tracking에서는 Warm-up이 그 역할을 합니다. 무엇보다 문제의 원인은 "대기 시간 부족"이 아니라 "오염된 지표"입니다.
Scheduled Scaling → 시간 기반. 문제 상황과 무관.
Max를 낮춘다 → 확장 자체를 막을 뿐 원인 해결이 아닙니다.
🎯 시험 팁 Cooldown = "방금 스케일링했으니 좀 쉬어" (시간을 막음)
Warm-up = "새 애 지표는 아직 믿지 마" (데이터를 거름)
Final Review

빠른 요약 & 시험 TIP

시험장 들어가기 직전 30초 스캔용.

⬆️➡️ 수직 vs 수평
타입/크기 변경 = 수직(Scale Up). 개수 변경 = 수평(Scale Out). ASG는 수평.
🌐 ELB는 static DNS
고정 IP는 NLB만. 일반 ELB는 변하지 않는 DNS 이름만 제공.
🔌 계층별 프로토콜
ALB=HTTP/HTTPS(L7), NLB=TCP/UDP/TLS(L4), GWLB=IP패킷/GENEVE 6081(L3).
📋 대상 그룹 = 명단
명단 + 헬스체크 규칙만 보관. 트래픽을 보내는 건 오직 로드밸런서.
⚖️ ASG ≠ 로드밸런싱
ASG는 개수만 관리. LB를 지우면 분산은 멈춘다. 둘은 완전히 다른 역할.
🌍 Cross-Zone 기본값
ALB=ON(무료). NLB/GWLB=OFF(켜면 AZ간 데이터 요금).
📜 SNI
리스너 1개에 인증서 여러 개. ALB/NLB/CloudFront 지원, CLB 미지원.
🚧 Max는 천장
Desired == Max면 CPU가 아무리 높아도 확장되지 않는다.
♻️ ASG의 복구
비정상 인스턴스를 고치지 않고 종료 후 새로 만든다.
⏳ Cooldown 300초
스케일링 직후 5분간 추가 스케일링 없음. 시간을 막는다.
🔥 Instance Warm-up
새 인스턴스를 지표 계산에서 제외. 데이터를 거른다. Target Tracking용.
📉 Target Tracking은 양방향
목표보다 높으면 확장, 낮으면 축소. 바닥은 Min, 천장은 Max.
🎲 NLB의 대상 선택
규칙 없음. 새 연결마다 flow hash 1회 계산 → 대상 고정.
⚡ NLB가 빠른 이유
TCP가 가벼워서가 아니라, 연결 종료·HTTP 파싱·규칙 매칭을 생략해서.
🩺 unhealthy 원인
거의 항상 ① SG 인바운드 ② 헬스체크 경로/상태코드. 이 둘.
🎯 시험 TIP 모음 — 키워드 → 정답
  • r4.large → r4.4xlarge → 수직 확장 (Vertical Scalability)
  • ASG로 인스턴스 개수 증감 → 수평 확장 (Horizontal Scalability)
  • ELB가 제공하는 것 → static DNS name (static IP 아님)
  • 고정 IP / 방화벽 화이트리스트 / 규제 승인 → NLB (+ Elastic IP)
  • 초당 수백만 요청 / 초저지연 → NLB
  • TCP · UDP · TLS → NLB (L4)
  • 경로 · 호스트명 · 헤더 · 쿼리로 라우팅 → ALB (L7)
  • Geolocation 기반 라우팅 → ALB 불가, Route 53
  • 방화벽 · IDS/IPS · GENEVE · 포트 6081 → GWLB (L3)
  • 로그인이 자꾸 풀린다 / 장바구니 사라짐 → Sticky Session 활성화
  • ALB 커스텀 쿠키 금지 이름 → AWSALB, AWSALBAPP, AWSALBTG
  • ALB 뒤에서 클라이언트 IP 알고 싶다 → X-Forwarded-For 헤더
  • 간헐적으로 접속 안 됨 / 인스턴스 충돌 → ELB Health Check
  • AZ간 트래픽 균등한데 특정 AZ만 CPU 높음 → Cross-Zone Load Balancing 활성화
  • 리스너 1개 + SSL 인증서 여러 개 → SNI
  • 여러 도메인 HTTPS + ALB → HTTPS 리스너 1개 + 인증서 다중 등록(SNI) + ACM
  • ALB를 NLB 대상으로 등록 → 가능 (반대는 불가)
  • Lambda를 대상 그룹에 등록 → ALB만 가능
  • Desired = Max 인데 CPU 80% → 아무 일도 일어나지 않음
  • ASG + ELB Health Check 비정상 → 종료 후 새 인스턴스 시작 (교체)
  • "평균 ~을 ~로 유지" → Target Tracking Scaling Policy
  • "임계값 넘으면 N대 추가" → Step / Simple Scaling Policy
  • AWS 기본 지표가 아닌 값으로 스케일링 → Custom Metric (PutMetricData) + CloudWatch Alarm
  • 사용 패턴을 미리 아는 경우 → Scheduled Scaling
  • 스케일링 후 5분(300초) 대기 → Cooldown Period
  • EC2 SG의 인바운드 소스 → LB의 보안 그룹 ID (IP 아님)
  • UDP 트래픽 (게임 서버, MQTT, DNS) → NLB (ALB는 즉시 탈락)
  • 클라이언트 IP 보존이 기본 → NLB (ALB는 X-Forwarded-For 필요)
  • NLB가 대상 고르는 방식 → Flow Hash (5튜플 + TCP 시퀀스, 연결당 1회)
  • NLB Sticky Session → Source IP Affinity (쿠키 아님, 쿠키는 HTTP 개념)
  • SSH로는 되는데 대상 그룹은 unhealthy → ① SG 인바운드 ② 헬스체크 경로/코드
  • Target Tracking + CPU가 목표보다 낮음 → Min까지 축소 (양방향 정책)
  • 부팅 중 CPU 100%가 평균 오염 → 과잉 확장 → Instance Warm-up
  • Cooldown vs Warm-up → Cooldown은 시간을 막고, Warm-up은 데이터를 거른다
  • 규제(PCI 등) 트래픽 분리 → ALB 호스트 규칙 + 대상 그룹 분리 (Sticky는 격리 수단 아님)
