🔐 IAM 개요
💡
IAM이란? AWS 계정의 인증(Authentication)과 인가(Authorization)를 관리하는 글로벌 서비스입니다. 모든 AWS 계정에 기본 포함되어 있으며 리전에 종속되지 않습니다.
항목내용
정식 명칭 Identity and Access Management
서비스 범위 글로벌(Global) — 특정 리전에 종속되지 않음
루트 계정 AWS 계정 최초 생성 시 만들어지는 최상위 계정. 일상 업무 사용 금지
핵심 역할 누가(Who) 무엇을(What) 어떤 조건에서(Condition) AWS 리소스에 접근할 수 있는지 정의
👥 IAM 구성요소

IAM은 4가지 핵심 구성요소로 이루어집니다. 각 요소의 특성과 차이점을 구분하세요.

Users

사용자

  • › 실제 사람 또는 서비스를 나타냄
  • › 여러 그룹에 동시에 속할 수 있음
  • › 그룹에 속하지 않아도 됨
  • › 사용자에게 직접 정책 부여 가능
Groups

사용자 그룹

  • › 사용자(User)의 집합
  • › 다른 그룹 포함 불가 ⚠️
  • › 그룹에 정책 연결 → 구성원 전체 적용
  • › 권한 관리의 기본 단위
Roles

역할

  • › AWS 서비스·외부 엔티티에 임시 권한 부여
  • › STS로 임시 자격 증명 발급
  • › EC2, Lambda 등에 부여
  • › Cross-account 접근에도 사용
Policies

정책

  • › 권한을 JSON 형식으로 정의
  • › 사용자·그룹·역할에 연결
  • › Effect: Allow 또는 Deny
  • › 명시적 Deny가 Allow보다 우선
⚠️
자주 나오는 함정: IAM 그룹에는 사용자(User)만 포함할 수 있습니다. 그룹 안에 다른 그룹을 중첩(Nested Group)하는 것은 불가능합니다.
📋 IAM 정책(Policy) 구조
🧠
기억법: 정책은 "문서(Document) 안에 여러 문(Statement)이 있는" 구조입니다. Version은 문서 최상단 레벨에만 존재하고, 나머지 요소들은 각 Statement 안에 있습니다.
요소위치설명예시
Version 문서 레벨 ⚠️ 정책 언어 버전. Statement에 포함되지 않음 "2012-10-17"
Statement 문서 레벨 하나 이상의 권한 문 배열 [{...}, {...}]
Sid Statement 내부 (선택) Statement 식별자 "AllowS3Read"
Effect Statement 내부 Allow 또는 Deny "Allow"
Principal Statement 내부 정책이 적용될 주체 "arn:aws:iam::..."
Action Statement 내부 허용/거부할 작업 "s3:GetObject"
Resource Statement 내부 대상 리소스 ARN "arn:aws:s3:::bucket/*"
Condition Statement 내부 (선택) 추가 조건 (IP, MFA 등) {"IpAddress": ...}

정책 종류 비교

AWS Managed

AWS 관리형 정책

  • › AWS가 미리 생성하고 관리
  • › 일반적인 사용 사례에 맞게 설계
  • › 여러 엔티티에 재사용 가능
  • › 예: AdministratorAccess
Customer Managed

고객 관리형 정책

  • › 고객이 직접 생성·관리
  • › 세밀한 권한 설정 가능
  • › 버전 관리 지원
  • › 여러 엔티티에 재사용 가능
Inline

인라인 정책

  • › 특정 사용자/그룹/역할에 1:1 연결
  • › 재사용 불가
  • › 대상 삭제 시 함께 삭제됨
🛡️ 공동 책임 모델 (Shared Responsibility Model)
구분 AWS 책임 — Security OF the Cloud 고객 책임 — Security IN the Cloud
인프라 물리적 데이터 센터 보안, 하드웨어 운영 —
네트워크 글로벌 네트워크 인프라 운영·유지보수 —
IAM 서비스 자체 IAM 서비스 운영·패치·가용성 보장 —
IAM 사용자 관리 — 사용자·그룹·역할 생성 및 관리
정책 설정 — IAM 정책 작성, 최소 권한 원칙 적용
계정 보안 — 루트 계정 MFA 활성화, 액세스 키 주기적 교체
💡
핵심 구분: AWS는 IAM 서비스 자체를 운영할 책임이 있지만, 그 안에서 누구에게 어떤 권한을 줄지는 전적으로 고객의 책임입니다.
🔍 IAM 보안 도구
도구분석 범위용도출력
Credentials Report
자격 증명 보고서		 계정(Account) 레벨 계정의 모든 IAM 사용자 자격 증명 상태 확인 — 비밀번호 사용 여부, MFA 활성화 여부, 액세스 키 현황 등 CSV 파일
Access Advisor
액세스 관리자		 사용자(User) 레벨 사용자에게 부여된 서비스 권한과 마지막 접근 시간 확인 — 불필요한 권한 식별에 활용 콘솔 화면
🧠
활용법: Access Advisor로 실제로 사용되지 않는 권한을 발견하고 제거하면 최소 권한 원칙(Least Privilege)을 자연스럽게 적용할 수 있습니다.
📝 시험 예상 문제

클릭하면 정답과 해설이 펼쳐집니다.

1
AWS Shared Responsibility Model에 따르면 다음 중 AWS의 책임은 무엇일까요?
공동 책임 모델
▶
정답
☁️ 클라우드 인프라 보안 — Security OF the Cloud
AWS는 IAM 서비스 자체의 운영·유지보수, 물리적 데이터 센터 보안, 하드웨어와 네트워크 인프라를 책임집니다.
반면 IAM 사용자·그룹·정책 생성, MFA 활성화, 액세스 키 관리 등은 고객 책임입니다.
🎯 시험 팁 "AWS 책임"이 보이면 → 물리적 인프라·서비스 운영. "고객 책임"이 보이면 → IAM 정책 설정, MFA, 키 관리
2
IAM 정책은 하나 이상의 Statement로 구성됩니다. Statement를 구성하는 요소가 아닌 것은?
정책 구조
▶
정답 (Statement에 포함되지 않는 요소)
❌ Version
Version은 정책 문서(Document) 레벨의 요소로, 개별 Statement 안에 포함되지 않습니다.
Statement의 구성요소: Sid(선택) · Effect · Principal · Action · Resource · Condition(선택)
🎯 시험 팁 "Statement에 포함되지 않는 것" → Version. Version은 항상 문서 최상단에 독립적으로 위치
3
루트 계정 보안을 높이려면 어떻게 해야 할까요?
보안
▶
정답
🔐 MFA (다단계 인증) 활성화
루트 계정에 MFA를 반드시 설정하고, 일상 업무에는 루트 계정을 사용하지 않아야 합니다.
루트 자격 증명을 절대 공유하지 않고, 루트 액세스 키는 생성하지 않는 것이 모범 사례입니다.
🎯 시험 팁 "루트 계정 보안" → MFA가 핵심 정답. 루트 계정은 초기 설정 이후 잠가두는 것이 원칙
4
다음 중 IAM 정책(Policy)의 종류에 해당하는 것은?
정책 종류
▶
정답
AWS Managed · Customer Managed · Inline Policy
세 가지 정책 유형 모두 JSON 형식으로 작성됩니다.
· AWS Managed: AWS가 미리 생성·관리. 재사용 가능
· Customer Managed: 고객이 직접 생성·관리. 세밀한 제어 가능
· Inline: 특정 엔티티에 1:1 연결. 대상 삭제 시 함께 삭제
🎯 시험 팁 재사용이 필요하면 Managed Policy. 1:1 연결로 재사용 불가 → Inline Policy
5
다음 중 IAM 모범 사례(Best Practice)는 무엇일까요?
모범 사례
▶
정답
✅ 최소 권한 원칙 적용 + 그룹으로 권한 관리
IAM 모범 사례 목록:
① 루트 계정 MFA 활성화
② 루트 계정 일상 사용 금지
③ 개인별 IAM 사용자 생성 (계정 공유 금지)
④ 그룹을 통한 권한 관리 (사용자에게 직접 정책 부여 지양)
⑤ 최소 권한 원칙(Least Privilege) 적용
⑥ 액세스 키 주기적 교체
⑦ Credentials Report로 정기 감사
🎯 시험 팁 "필요한 것 이상의 권한을 주지 말라" = Least Privilege. 사용자에 직접 정책 부여보다 그룹 경유가 모범 사례
6
다음 중 IAM 사용자(User)에 관한 설명으로 잘못된 것은?
사용자
▶
정답 (잘못된 설명)
❌ "IAM 사용자는 반드시 하나의 그룹에만 속해야 한다"
올바른 사실: IAM 사용자는 여러 그룹에 동시에 속할 수 있습니다.
또한 어떤 그룹에도 속하지 않아도 되며, 사용자에게 직접 정책을 부여하는 것도 가능합니다.
사용자의 최종 권한 = 그룹 정책 합산 + 직접 부여 정책
🎯 시험 팁 사용자는 0개~여러 개의 그룹에 소속 가능. "하나의 그룹에만" 이라는 표현이 나오면 오답
7
다음 중 IAM 보안 도구는 무엇일까요?
보안 도구
▶
정답
📊 Credentials Report + Access Advisor
Credentials Report: 계정 레벨. 모든 IAM 사용자의 자격 증명 현황(비밀번호·MFA·액세스 키 상태)을 CSV로 출력합니다.

Access Advisor: 사용자 레벨. 사용자에게 부여된 서비스 권한과 마지막 접근 시간을 보여줘 불필요한 권한 제거에 활용합니다.
🎯 시험 팁 계정 전체 현황 → Credentials Report / 특정 사용자 권한 검토 → Access Advisor
8
IAM 사용자 그룹에는 IAM 사용자 및 기타 사용자 그룹이 포함될 수 있다. (True / False)
O/X 문제
▶
정답
❌ False (거짓)
IAM 그룹에는 IAM 사용자(User)만 포함할 수 있습니다. 다른 그룹을 그룹 안에 넣는 중첩(Nested Group)은 지원되지 않습니다.
· 그룹 → 사용자 ✅
· 그룹 → 그룹 ❌
🎯 시험 팁 "그룹에 그룹 포함 가능하다" → 무조건 False. IAM 그룹의 구성원은 오직 사용자뿐
9
IAM 권한을 부여할 때 어떤 원칙을 적용해야 할까요?
원칙
▶
정답
⚖️ 최소 권한 원칙 — Principle of Least Privilege
작업 수행에 필요한 최소한의 권한만 부여해야 합니다. 처음에는 좁은 권한으로 시작하고, 필요에 따라 점진적으로 추가하는 것이 올바른 방법입니다.
Access Advisor로 실제 사용 현황을 주기적으로 검토해 불필요한 권한을 제거합니다.
🎯 시험 팁 "권한 부여 원칙", "보안 원칙" 키워드 → Least Privilege. 권한은 처음부터 넓게 주지 말고 필요한 것만
10
IAM 역할(Role)에 대한 올바른 설명은?
역할
▶
정답
🎭 AWS 서비스·외부 엔티티에 임시 권한을 부여하는 메커니즘
IAM 역할은 사람이 아닌 AWS 서비스나 외부 사용자에게 임시 보안 자격 증명(STS)을 통해 권한을 부여합니다.
예: EC2 인스턴스가 S3에 접근해야 할 때, EC2에 S3 읽기 역할을 부여합니다.
영구적인 액세스 키 없이 안전하게 AWS 서비스 간 권한을 위임할 수 있습니다.
🎯 시험 팁 "EC2/Lambda가 다른 AWS 서비스 접근" → IAM Role 사용. "Cross-account 접근" → IAM Role. 사람이 아닌 서비스에 권한을 줄 때 항상 Role

🎯 시험 TIP 모음
  • 루트 계정 보안 강화 → MFA 활성화가 핵심 정답
  • Statement에 포함되지 않는 요소 → Version (문서 레벨 요소)
  • 그룹에 그룹 포함 가능하다 → 무조건 False
  • EC2 / Lambda → 다른 AWS 서비스 접근 → IAM Role 사용
  • Cross-account 접근 → IAM Role 사용
  • 계정 전체 자격 증명 현황 → Credentials Report
  • 사용자별 서비스 접근 이력 → Access Advisor
  • 권한 부여 원칙 → Least Privilege (최소 권한)
  • AWS 책임 (Shared Responsibility) → 인프라·서비스 운영 (정책 설정은 고객 책임)
